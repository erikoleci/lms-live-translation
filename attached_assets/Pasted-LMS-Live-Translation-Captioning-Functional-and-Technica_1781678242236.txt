LMS Live Translation & Captioning — Functional and Technical Specification
1. Objective
Build a stripped-down Wordly-like module for academic use inside TILI, enabling a teacher to start a live session where students join by QR code and receive real-time captions, translations, and translated audio on their own devices.
Initial languages: Italian, English, Albanian.	
Core stack: Frontend: Vue 3, Vite, Vuetify. Backend: Java Quarkus. Database: PostgreSQL. Realtime transport: WebSocket/WebRTC depending on provider integration. AI engine: pluggable provider architecture supporting cloud APIs and local models.
2. User Roles
Teacher: Can create, start, pause, resume, and end a session. Can select source language or enable auto-detection. Can share QR code/session link. Can monitor participant count. Can optionally download transcript after session.
Student/Participant: Can join via QR code without mandatory login, or with LMS SSO if required. Can choose display language: original, Albanian, English, Italian. Can view live captions. Can enable translated audio playback. Can select from available TTS voices. Can change language during session.
Admin: Can configure providers for STT, translation, and TTS. Can define enabled languages and voices. Can configure retention policy, cost limits, and privacy settings. Can view usage statistics.
3. Main Functional Flow
Teacher opens LMS course page and clicks “Start Live Translation Session”.
System creates a session with: unique session ID, short join code, QR code, teacher ID, course ID, source language, enabled target languages, status = waiting/active.
Teacher starts audio capture from browser microphone.
Audio is streamed to backend or directly to AI provider using ephemeral credentials.
Backend receives transcription events and stores normalized segments.
For every finalized or partial transcript segment: system keeps original transcript, translates segment into enabled target languages, broadcasts text updates to connected students, optionally generates TTS audio for selected languages/voices.
Student scans QR code, joins session, selects language and audio preference.
Student receives: live original captions, translated captions, translated audio stream or audio chunks.
Teacher ends session.
System closes realtime pipeline, finalizes transcript, and optionally generates session export.
4. Core Features
4.1 Session Management
Required: Create session. Start session. Pause/resume audio processing. End session. Generate QR code. Generate public join URL. Limit session to LMS course context. Show live participant count. Expire inactive sessions automatically.
Recommended session states: CREATED, WAITING, ACTIVE, PAUSED, ENDED, FAILED, EXPIRED.
Each teacher can configure session access mode:
OPEN:
Anyone with the QR code or join link can participate.
CLOSED:
Only authenticated users can join.
Session settings:
accessMode: OPEN / CLOSED
recordingEnabled: true / false
studentTranscriptDownloadEnabled: true / false
maxParticipants: default 300
sourceLanguage: IT / EN / SQ
targetLanguages: IT / EN / SQ
4.2 Audio Capture
Teacher browser must capture microphone audio using Web Audio API / MediaDevices API.
Required: microphone permission handling, audio level indicator, mute/unmute, reconnect on network interruption, detect no-audio condition, warn teacher if input is silent.
Preferred browser format: Opus/WebM or PCM16 depending on STT provider.
Audio recording must be supported but optional.
If enabled:
teacher audio is recorded server-side,
recording is linked to the session,
recording can be retained or deleted based on policy,
recording must not be available to students unless explicitly allowed.
If disabled:
audio is processed live only,
no raw audio is stored,
only transcript/translation segments may be stored.
4.3 Speech-to-Text
The STT component must receive teacher audio and produce: partial transcript segments, final transcript segments, language detection metadata where available, confidence score where available, timestamps.
Initial source languages: Italian, English, Albanian.
Provider options: Cloud preferred for MVP: OpenAI Realtime Transcription / GPT realtime whisper. Azure Speech to Text. AWS Transcribe. Google Speech-to-Text.
Open-source/local: Whisper. whisper.cpp. faster-whisper. Vosk, only if language quality is acceptable after testing.
Recommended MVP: Use OpenAI Realtime transcription or Azure Speech for lower integration complexity and better real-time behavior. Keep Whisper/faster-whisper as local/private deployment option.
4.4 Translation
The translation engine must translate transcript segments from source language into: Albanian, English, Italian.
Required: preserve academic terminology where possible, handle partial segments without creating excessive flicker, re-translate final segments when the STT final result differs from the partial result, support glossary/custom terminology in later phase.
Provider options: Cloud: OpenAI text model or realtime translation. Azure Translator. Google Cloud Translation. AWS Translate. DeepL API, if Albanian support and licensing are acceptable.
Open-source/local: NLLB, MarianMT, M2M100, LibreTranslate as wrapper, subject to quality testing.
Recommended MVP: Use OpenAI or Azure for translation. Add local NLLB/LibreTranslate adapter later.
4.5 Text-to-Speech
Students can listen to translated audio in their selected language.
Required: TTS from translated text, voice selection from predefined voices, AI-generated voice disclosure in UI, play/pause audio, audio queue per student, avoid overlapping audio chunks, option to disable audio.
Provider options: Cloud: OpenAI TTS. Azure Neural TTS. AWS Polly. Google Text-to-Speech.
Open-source/local: Piper TTS. Coqui TTS, if maintained/deployment risk is acceptable. WhisperSpeech, experimental.
Recommended MVP: Use OpenAI TTS or Azure Neural TTS. Use Piper as local fallback for selected languages only after voice-quality testing.
4.6 On Premise AI Engine
Recommended local stack (not for MVP, but for future support):
Speech-to-text:
faster-whisper with Whisper large-v3 or medium model.
GPU recommended for acceptable live latency.
Translation:
NLLB or MarianMT for open-source translation.
For MVP, prioritize English ↔ Italian quality.
Albanian can be enabled with lower acceptance expectations.
Text-to-speech:
Piper TTS for local neural voices.
Alternative: Coqui/Piper-compatible voice models if quality is insufficient.
Recommended architecture:
Quarkus does not run AI inference directly.
Quarkus calls local AI worker services over internal HTTP/gRPC/WebSocket.
Suggested services:
stt-worker
translation-worker
tts-worker

5. Architecture
5.1 High-Level Components
Vue Frontend: Teacher session UI. Student join UI. Live captions UI. Audio playback UI. Admin configuration UI.
Quarkus Backend: REST API. WebSocket gateway. Session manager. Audio stream coordinator. AI provider abstraction. Translation orchestrator. TTS orchestrator. Transcript storage. LMS integration layer.
AI Worker Layer: May run inside backend initially. Can later be split into separate worker service. Responsible for STT, translation, and TTS jobs.
Database: PostgreSQL for sessions, participants, transcripts, provider config, usage logs.
Object Storage: Optional for recorded audio and generated TTS chunks. Prefer disabled audio recording by default for privacy.
5.2 Recommended Runtime Architecture
For MVP: Vue SPA → Quarkus REST/WebSocket → Cloud AI APIs → PostgreSQL.
For production: Vue SPA behind LMS or reverse proxy. Quarkus backend with horizontal scaling. Redis for session pub/sub and ephemeral state. PostgreSQL for durable state. Object storage for optional exports. AI workers separated from API nodes if load grows.
6. Backend Specification — Quarkus
Recommended extensions: quarkus-rest-jackson, quarkus-hibernate-orm-panache, quarkus-jdbc-postgresql, quarkus-websockets-next, quarkus-smallrye-openapi, quarkus-oidc, quarkus-scheduler, quarkus-cache, quarkus-redis-client, quarkus-micrometer, quarkus-smallrye-health.

Events: SESSION_STATUS_CHANGED, PARTICIPANT_JOINED, PARTICIPANT_LEFT, TRANSCRIPT_PARTIAL, TRANSCRIPT_FINAL, TRANSLATION_PARTIAL, TRANSLATION_FINAL, TTS_AUDIO_CHUNK_READY, ERROR, HEARTBEAT.
6.3 AI Provider Abstraction
Define interfaces:
SpeechToTextProvider: startSession(config) sendAudioChunk(sessionRef, chunk) onPartialTranscript(callback) onFinalTranscript(callback) closeSession(sessionRef)
TranslationProvider: translate(text, sourceLang, targetLang, glossaryContext)
TextToSpeechProvider: synthesize(text, language, voice, format) stream(text, language, voice)
ProviderConfig: providerCode, providerType: STT / TRANSLATION / TTS, priority, enabled, credentialsRef, supportedLanguages, supportedVoices, costLimit, timeoutMs, fallbackProviderCode.
The backend must support provider fallback: Primary STT fails → fallback STT. Primary translation fails → fallback translation. Primary TTS fails → disable audio and keep captions active.
7. Frontend Specification — Vue
Recommended libraries: Vue 3 Composition API. Vite. Vuetify 3. Pinia. Vue Router. Axios. Native WebSocket API or socket wrapper. qrcode.vue for QR rendering. MediaDevices/Web Audio API for microphone capture. wavesurfer.js optional for audio visualization.
7.1 Teacher UI
Pages: Live Session Dashboard. Create Session Modal. Active Session View. Transcript Export View.
Teacher Active Session must show: session title, course name, QR code, join URL, source language, session status, participant count, microphone status, audio level meter, start/pause/end buttons, live original transcript, error/warning messages.
7.2 Student UI
Pages: Join Session. Language Selection. Live Captions. Audio Settings.
Student Live View must show: selected language, caption stream, toggle original/translated captions, audio on/off, voice selector, font size selector, dark/light mode, connection status.
Accessibility: large text mode, high contrast mode, keyboard navigation, screen-reader-friendly labels.
8. Data Model
live_session: id, title, teacher_id, access_mode, source_language, target_languages, status, join_code, recording_enabled, student_transcript_download_enabled, max_participants, created_at, started_at, ended_at, expires_at.
audio_recording: id, session_id, file_path, format, duration_ms, size_bytes, retention_until, created_at, deleted_at.
live_participant: id, session_id, user_id nullable, anonymous_name, target_language, audio_enabled, voice_code, joined_at, left_at, connection_status.
transcript_segment: id, session_id, sequence_no, source_language, original_text, is_final, confidence, start_offset_ms, end_offset_ms, created_at.
translation_segment: id, transcript_segment_id, target_language, translated_text, is_final, created_at.
tts_chunk: id, translation_segment_id, language, voice_code, file_path nullable, duration_ms, created_at.
9. Realtime Processing Logic
Audio chunking: teacher browser sends small chunks every 100–500 ms, depending on provider. Backend assigns monotonic sequence number. Backend forwards chunks to STT provider.
Transcript handling: partial segments are broadcast quickly. final segments replace previous partial segments. Every segment must include sequence number and timestamp.
Translation handling: for partial text, translate with debounce. for final text, always translate again. Cache identical source text + language pair to reduce cost.
TTS handling: generate only for languages currently requested by active students. Do not pre-generate all voices unless configured. Queue audio chunks per participant. Prioritize final segments over partial segments for TTS.
Latency targets: caption partial visible under 2 seconds. final caption under 4 seconds. translated caption under 5 seconds. translated audio under 7 seconds.
10. LMS Integration
MVP: no Integration necessary.
LMS SSO. Course ID mapped to live_session.course_id.
Preferred: OIDC SSO through existing Keycloak identity provider. Teacher authorization from LMS role. Student join either anonymous QR or authenticated LMS user.
Integration requirements: only teachers of the course can start sessions, students can join sessions only for their course unless anonymous access is enabled, session transcript can be attached to the LMS course after completion.
11. Security and Privacy
Do not store raw audio by default. Store transcripts only if enabled by teacher/admin. Apply retention policy, for example 30/90/180 days. Use HTTPS only. Use WebSocket over TLS only. Use short-lived join tokens. Use ephemeral provider credentials for browser-direct AI sessions. Do not expose permanent AI API keys to frontend. Log provider usage without storing sensitive audio. Add visible disclosure when AI-generated audio is played. Add consent notice before students join. Allow teacher/admin to disable transcript export.
12. Admin Configuration
Admin can configure: enabled source languages, enabled target languages, STT provider priority, translation provider priority, TTS provider priority, available voices per language, session max duration, max participants per session, recording/transcript retention, cost budget per session, fallback behavior, LMS integration settings.
13. Suggested MVP Scope
MVP should include: teacher starts session, QR join, student language selection, live captions, live translation, OpenAI or Azure cloud provider integration, PostgreSQL persistence, session transcript export, basic admin provider configuration.
Defer to Phase 2: local Whisper/faster-whisper deployment, local NLLB translation, local Piper TTS, glossary support, speaker diarization, lecture summaries, LMS gradebook/activity integration, multi-teacher sessions, recorded lecture upload translation, advanced cost analytics.
14. Recommended Provider Strategy
MVP provider profile: STT: OpenAI Realtime Transcription or Azure Speech. Translation: OpenAI or Azure Translator. TTS: OpenAI TTS or Azure Neural TTS.
Local/private profile: STT: faster-whisper or whisper.cpp. Translation: NLLB or LibreTranslate. TTS: Piper.
Enterprise/cloud fallback profile: STT: AWS Transcribe. Translation: AWS Translate. TTS: AWS Polly.
15. Acceptance Criteria
Teacher can create and start a session from LMS. QR code opens student join page. Student can select Albanian, English, or Italian. Teacher speech appears as original captions. Translated captions appear for selected language. Student can enable translated audio. Admin can switch provider configuration without code change. Session can be ended cleanly. Transcript can be exported if retention/export is enabled. System handles provider failure gracefully. No permanent AI API key is exposed in frontend.
